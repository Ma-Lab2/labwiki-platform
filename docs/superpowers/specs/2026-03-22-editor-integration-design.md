# 编辑器联动版设计

## 背景

当前知识助手已经支持在私有 wiki 的源码编辑页中：

- 基于当前页面生成草稿
- 在用户确认后把草稿填入 `wpTextbox1`
- 不自动保存页面

但这只覆盖 `action=edit` 的源码编辑入口。私有 wiki 实际还存在：

- `PageForms` 表单编辑
- `VisualEditor` 富文本编辑

三者的 DOM、状态管理和适合的填充方式不同，不能按“统一往页面里灌文本”处理。

## 目标

按优先级扩展“知识助手填充”能力：

1. 保留并稳固源码编辑直填
2. 为 `PageForms` 提供字段级填充
3. `VisualEditor` 第一阶段只提供草稿生成与跳转源码编辑，不做直接注入

## 设计决策

### 1. 源码编辑

保持当前路径：

- 页面识别：存在 `#wpTextbox1`
- 入口：编辑页顶部“知识助手填充”
- 流程：
  - 打开抽屉
  - 预填“整理当前页面为可直接填入编辑框的草稿”
  - 生成草稿
  - 用户确认
  - 填入 `wpTextbox1`
  - 页面仍停留在编辑态，由用户手动保存

### 2. PageForms

第一阶段目标是“字段级建议 + 一键填入”，不自动提交表单。

页面识别：

- `wgCanonicalSpecialPageName === 'FormEdit'`
- 或 `wgAction === 'formedit'`
- 或页面中存在 `#pfForm`

交互方式：

- 在 `#pfForm` 顶部注入“知识助手填表”入口
- 打开抽屉并预填表单导向问题
- 助手返回结构化字段建议
- 前端做字段匹配：
  - 识别字段标签、输入名、textarea/select
  - 生成“可填字段清单”
- UI 提供：
  - “填入此字段”
  - “一键填入全部可匹配字段”

边界：

- 只填可识别字段
- 不自动提交表单
- 不碰复杂 widget 的内部状态，除非确认存在标准 input/textarea/select

### 3. VisualEditor

第一阶段不做内容注入。

只提供：

- 在 VE 页面打开知识助手
- 生成页面草稿
- 给出“切换到源码编辑并填入”入口

原因：

- VE 是富文本状态机，直接注入成本高、脆弱且回归面大
- 当前产品目标是“安全可用”，不是“完全接管任意编辑器”

## 页面类型优先级

实现顺序固定为：

1. 源码编辑完善
2. PageForms 字段级填充
3. VisualEditor 草稿辅助

## 风险与约束

- `PageForms` 字段命名可能不统一，需要先做保守匹配，不做激进推断
- 复杂 `PageForms` widget 可能需要后续专门支持，第一阶段只覆盖原生表单控件
- `VisualEditor` 不直接注入，避免破坏编辑状态
- 所有入口都遵守“不自动保存/不自动提交”

## 验收标准

### 源码编辑

- 编辑页顶部出现助手填充入口
- 草稿可填入 `wpTextbox1`
- 页面保持在 `action=edit`

### PageForms

- `FormEdit` 页面顶部出现助手填表入口
- 助手返回结构化字段建议
- 至少一组表单页面可实现字段匹配和一键填入
- 表单不自动提交

### VisualEditor

- VE 页面可打开助手
- 可生成草稿
- 明确引导用户切到源码编辑继续填入
